# Contributing to Everything About Loop Engineering

Thank you for considering a contribution. This is a young, fast-moving field — the most valuable contributions right now are corrections, missing context, and real-world experience reports.

## How to Contribute

### Reporting Issues

- **Factual errors**: Open an issue with the module name, the incorrect claim, and a source (URL, talk transcript, etc.) that contradicts it.
- **Missing content**: Open an issue describing what's absent and why it matters.
- **Broken links**: Open an issue with the module and the broken URL.

### Submitting Changes

1. Fork the repository.
2. Create a branch (`git checkout -b fix/module-03-typo`).
3. Make your change.
4. Run a quick self-check: does the markdown render? Are links valid? Are Mermaid diagrams syntactically correct?
5. Open a pull request with a clear title and description of what changed and why.

### Content Guidelines

- **Accuracy over completeness.** If you're unsure about a claim, say so in the text rather than asserting it.
- **Attribute everything.** Quotes, statistics, and case studies must have sources. If a source is self-reported, say "self-reported" every time.
- **Plain English first, technical detail second.** Every module follows this structure. Don't break it.
- **Beginner-friendly in early modules.** Modules 00–02 assume zero coding-agent experience. Don't introduce jargon without defining it.
- **No invented facts.** Do not fabricate metrics, view counts, quotes, or case studies. Use only the source data provided in the project or verifiable public sources.

### Style

- Use GitHub-flavored Markdown.
- Mermaid diagrams should be self-contained (no external dependencies).
- Keep files under 800 lines.
- Use `Title Case` for H1/H2 headings, `sentence case` for H3+.
- Cross-reference other modules and the glossary when using defined terms.

### Code of Conduct

Be respectful. This is an educational project. Assume good intent. Disagree with ideas, not people.
