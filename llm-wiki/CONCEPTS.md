# LLM Wiki: Core Concepts

> Plain-English and technical definitions of every core concept in loop engineering.

---

## Loop Engineering

**Plain English:** Replacing yourself as the person who prompts an AI coding agent — designing the system that does it instead.

**Technical:** The practice of designing automated systems that prompt, verify, and iterate on AI coding agent work, using scheduling, state management, sub-agent verification, and external memory to enable autonomous operation.

**Key quote:** "Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go." — Addy Osmani

**Coined by:** Addy Osmani (June 8, 2026), building on Boris Cherny and Peter Steinberger

**Origin:** The term appeared in Osmani's Substack essay "Loop Engineering" on June 8, 2026, the same week Peter Steinberger's post about designing loops reached 2M–6.5M views on X.

---

## Coding Agent

**Plain English:** An AI that can take actions in the real world — read files, edit code, run commands — rather than just generating text.

**Technical:** An AI system with tool access (filesystem, terminal, APIs) that can autonomously execute multi-step tasks. Differs from a chatbot in that it can observe state, take actions, and iterate based on results.

**Examples:** Claude Code (Anthropic), Codex (OpenAI), Cursor, Windsurf

**Key characteristic:** Can run in a loop — observe, decide, act, verify, repeat.

---

## Prompt

**Plain English:** A text instruction given to an AI model.

**Technical:** Natural language input that instructs an LLM to perform a task. In coding agents, prompts can include task descriptions, context files, error messages, and feedback from previous iterations.

**In loop engineering:** Prompts are generated by the system, not typed by a human. The system decides what to prompt based on state, schedule, and verification results.

---

## Prompt Engineering

**Plain English:** The skill of writing good instructions for AI models.

**Technical:** The practice of crafting prompts that reliably produce desired behavior from LLMs. Includes techniques like few-shot examples, chain-of-thought reasoning, constraint specification, and context windowing.

**Relationship to loop engineering:** Prompt engineering is the input; loop engineering is the system that decides when and how to use those prompts.

---

## Control Loop

**Plain English:** A system that watches a condition and acts to maintain it — like a thermostat.

**Technical:** A feedback loop that continuously:
1. Observes current state
2. Compares to desired state
3. Takes action if there's a difference
4. Waits and repeats

**The shape:** This pattern is decades old — Kubernetes controllers, CI retry systems, TCP congestion control, and thermostats all use it. Loop engineering applies it to AI coding agents.

**Key insight:** The shape isn't new. The integration with coding agents (native scheduling, worktrees, sub-agents, memory) is what's new as of mid-2026.

## Ralph Loop

**Plain English:** The original loop technique — an infinite bash loop piping a prompt file to a coding agent.

**Technical:**
```bash
while : ; do cat PROMPT.md | claude ; done
```

**Created by:** Geoffrey Huntley (July 2025)

**Key characteristics:**
- Fresh context every iteration (solves the context window degradation problem)
- State persists in files and git, not in model memory
- "Deterministically bad in an undeterministic world" — reliable because it's simple
- One task per loop iteration

**Evolution:** Huntley ran this unattended for 3 months; the agent built CURSED (an esoteric programming language). In January 2026, he published "everything is a ralph loop." By mid-2026, Anthropic formalized it as a Claude Code plugin.

## Automations

**Plain English:** The "on switch" for your loop — a schedule or trigger that makes it run without you.

**Technical:** Mechanisms that initiate loop execution:
- **External scheduling:** Cron jobs, GitHub Actions workflows
- **Built-in scheduling:** Native features in Claude Code, Codex
- **Event-driven:** Triggered by PR creation, test failure, issue creation

**Key primitive:** Goal-condition verification — a separate model checks after each turn whether a stated condition is actually true, preventing the acting model from grading itself.

**Failure mode if skipped:** Loop only runs when you remember to start it manually.

---

## Worktrees

**Plain English:** Giving each agent its own copy of the project to work on.

**Technical:** Git worktrees that check out multiple branches simultaneously in different directories. Each worktree has its own working directory but shares the same git history.

**Purpose:** Prevents parallel agents from colliding on the same files. Essential when running multiple loops on the same repository.

**Native support:** Both Claude Code and Codex support worktrees natively.

**Failure mode if skipped:** Parallel agents overwrite each other's changes or create merge conflicts.

---

## Skills

**Plain English:** A document that tells the agent the rules of the project — so it doesn't have to figure them out from scratch every time.

**Technical:** A folder (conventionally `SKILL.md` plus supporting files) containing:
- Project conventions (language, framework, build tools)
- Build steps
- Things not to do (and why)
- Usage description (tight, boring descriptions beat clever ones for agent matching)

**Often paired with:**
- `VISION.md` — what the project is building toward
- `AGENTS.md` — house rules for agent behavior

**Failure mode if skipped:** Agent re-derives project context every run, wasting tokens and making inconsistent decisions.

---

## Plugins & Connectors

**Plain English:** Letting the agent reach outside the filesystem — to issue trackers, databases, chat, deployment.

**Technical:** Extensions built on MCP (Model Context Protocol) that provide standardized interfaces between agents and external tools.

**Common plugins:** Issue tracker (GitHub, Jira), Database, Chat (Slack, Discord), Deployment, CI/CD

**Failure mode if skipped:** Loop can only read/write files — no access to real-world systems.

---

## Sub-Agents

**Plain English:** A second agent that checks the first agent's work — like having a separate person grade the exam instead of letting the student grade their own.

**Technical:** An independent agent (sometimes a different model) that verifies the primary agent's output. Implements the maker/checker pattern:
- **Maker:** Does the work
- **Checker:** Independently verifies the work

**Why separate:** Different models have different blind spots. The checker can catch errors the maker misses. Self-assessment is unreliable.

**Failure mode if skipped:** Agent grades its own work — errors slip through invisibly.

---

## Memory & State

**Plain English:** A file that remembers what happened between agent runs — because the model itself forgets everything.

**Technical:** A durable file (conventionally `STATE.md`) outside any single conversation that persists:
- What was tried
- What passed
- What's still open
- When the last run happened

**Why it matters for loops:** Without state, every loop run starts from zero — no continuity, repeated work, repeated mistakes.

**Failure mode if skipped:** Loop forgets everything between runs, never learning or progressing.

## Maturity Tiers

### L1 (Report Only)

**What it does:** Reads, observes, writes down — changes nothing.

**When to use:** Most loops, most of the time. This is not a stepping stone — it's a destination for many tasks.

**Advice:** Stay at L1 far longer than feels necessary.

### L2 (Assisted / Patch-Only)

**What it does:** Proposes narrow-scope changes (dependency bumps, lint fixes, changelog drafts); human merges.

**When to use:** Only after L1 has been running successfully for at least 1 week, the scope is proven narrow, and failure modes are understood.

### L3 (Unattended)

**What it does:** Commits, merges, or deploys autonomously inside guardrails proven at L1/L2.

**When to use:** Rarely. Few tasks ever earn L3. The bar should be extremely high: weeks of successful L1 and L2, proven automated verification, minimal and reversible cost of wrong changes.
