# LLM Wiki: Terminology

> Every term used in this repository, alphabetized, with definitions and usage notes.

---

## A

**Agent** — An AI system that can take actions (editing files, running commands) rather than only generating text. Contrast with a chatbot, which only generates text.

**AGENTS.md** — A project-level markdown file defining house rules for agent behavior. Covers conventions, forbidden actions, coding style, and review requirements. Placed at the repo root.

**Automation** — A scheduled or triggered run that turns a single agent session into an ongoing loop. May be cron, GitHub Actions, built-in scheduling, or event-driven.

---

## B

**Building Block** — One of the six core components of a loop engineering system: automations, worktrees, skills, plugins & connectors, sub-agents, and memory/state.

---

## C

**Cadence** — How frequently a loop runs. Examples: once daily, every 5–15 minutes, every 6 hours. Key decision in loop design.

**Claude Code** — Anthropic's AI coding agent with native support for worktrees, sub-agents, and `/ralph-loop`. One of the two most-used coding agent products as of mid-2026.

**Control Loop** — A system that continuously monitors a condition and takes action to maintain it. The foundational pattern behind loop engineering. Decades old; the integration with coding agents is new.

---

## D

**Design Canvas** — A structured worksheet for planning a loop. Captures goal, verification conditions, building blocks, cadence, maturity tier, kill switch, and owner.

---

## G

**Goal-Condition Primitive** — An in-session check where a separate model verifies whether a stated condition is actually true after each agent turn. Prevents the acting model from grading itself.

---

## H

**HONESTY.md** — A project-level file documenting limitations, claim classification, and disclaimers.

---

## K

**Kill Switch** — A mechanism to immediately stop a running loop. May be a file-based flag, environment variable, or process signal. Must be tested before enabling any unattended loop.

---

## L

**L1 (Report Only)** — Lowest maturity tier. Loop reads, observes, writes down — changes nothing. Most loops should stay here.

**L2 (Assisted / Patch-Only)** — Middle maturity tier. Loop proposes narrow-scope changes; human reviews and merges.

**L3 (Unattended)** — Highest maturity tier. Loop commits/merges/deploys autonomously inside proven guardrails. Few tasks earn this.

**LLM** — Large Language Model. The AI model underlying coding agents (GPT, Claude, Gemini, etc.).

---

## M

**Maker/Checker** — Pattern where one agent (maker) produces work and a second, independent agent (checker) verifies it. Prevents self-grading.

**MCP** — Model Context Protocol. Standard for connecting agents to external tools (issue trackers, databases, chat, deployment).

**Memory/State** — A durable file outside any single conversation that persists information across agent runs. Typically STATE.md.

---

## P

**Plugin** — An extension built on MCP that lets an agent reach real tools beyond the filesystem.

**PR Babysitter** — Production pattern: loop monitors pull requests every 5–15 minutes for review readiness, CI status, and merge conflicts. Starts at L1.

**Prompt** — A text instruction given to an AI model. In loop engineering, prompts are generated by the system, not typed by a human.

**Prompt Engineering** — The practice of writing effective prompts. Loop engineering is the system that generates prompts automatically.

---

## R

**Ralph Loop** — The original loop technique by Geoffrey Huntley: `while : ; do cat PROMPT.md | claude ; done`. Fresh context every iteration, state in files.

---

## S

**SKILL.md** — A markdown file documenting project conventions, build steps, coding style, and things not to do. Prevents agents from re-deriving context.

**Sub-Agent** — A second, independent agent that verifies the primary agent's work. May use a different model. Implements maker/checker pattern.

---

## T

**TDD** — Test-Driven Development. Writing tests before implementation. In loop engineering, provides natural verification: a loop can run tests to check its output.

---

## V

**VISION.md** — A project-level file describing what agents should be building toward. Anchors multiple agent runs on the same objective.

---

## W

**Worktree** — An isolated working directory on its own git branch, sharing repo history. Prevents parallel agents from colliding on the same files.
